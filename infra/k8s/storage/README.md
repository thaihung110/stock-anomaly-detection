# Storage Infrastructure

This directory manages the foundation storage layer on Kubernetes: **MinIO** (S3-compatible object storage), **PostgreSQL** (relational metadata), **Keycloak** (OAuth2/OIDC auth), and **Gravitino** (Iceberg REST catalog). Every other layer (`orchestration/`, `compute/`) depends on this one being up first.

## Directory Structure

### `/config`

- `minio.yaml` — Helm values for MinIO (overrides default chart values)
- `postgresql.yaml` — Helm values for PostgreSQL (overrides default chart values)
- `keycloak.yaml` — Helm values for Keycloak (overrides default chart values)
- `gravitino.yaml` — Helm values for Gravitino (overrides the vendored chart in `helm/gravitino`)
- `storageclass.yaml` — the `hostpath` StorageClass definition used by every PVC in this project

### `/debug`

Point-in-time `kubectl describe`/event captures kept for troubleshooting reference: `event_keycloak.yaml`, `event_minio.yaml`, `event-nginx.yaml`.

### `/helm`

- `gravitino/` — a locally vendored copy (chart version `1.1.17`) of the official Apache Gravitino chart, pulled from `oci://registry-1.docker.io/bitnamicharts/gravitino`. Installed from here instead of a remote repo.

### `/scripts`

Automation scripts for installing, uninstalling, and configuring each component.

### `/test_template`

Template output for testing Helm chart rendering without deploying: `minio_template.yaml`, `postgresql_template.yaml`, `keycloak_template.yaml`, `gravitino_template.yaml`.

### `/tls`

Self-signed TLS certs generated by the `create_secret_*.sh` scripts: `keycloak_tls.{cert,key}`, `gravitino_ingress_tls.{cert,key}`. Also contains `lakekeeper_tls.{cert,key}` — leftover from before this project switched its Iceberg catalog from Lakekeeper to Gravitino; no longer used.

## Prerequisites: StorageClass & Ingress-NGINX

Every component below needs these two things in place first.

### 1. StorageClass (`hostpath`, Retain policy)

```bash
./scripts/set_storageclass_retain.sh
```

`config/storageclass.yaml` defines a StorageClass named `hostpath` (provisioner `docker.io/hostpath`, set as the cluster's default class). The script deletes and recreates this StorageClass with `reclaimPolicy: Retain`, then patches any existing PVs already on `hostpath` to `Retain` too — so a PVC deletion doesn't silently destroy the underlying data. Every component in this project (MinIO, PostgreSQL, Keycloak, Kafka, Airflow, Trino...) provisions its PVCs on this same `hostpath` class.

### 2. Ingress-NGINX Controller

```bash
./scripts/ingress.sh
```

This wipes any existing `ingress-nginx` deployment/service/namespace, then applies the official `ingress-nginx` controller manifest (`controller-v1.13.0`, cloud provider variant). This cluster runs on **kubeadm** — after applying, expose the controller however your kubeadm setup routes external traffic (NodePort, `hostNetwork`, MetalLB, etc.); no `minikube tunnel` step is needed here.

## Installing PostgreSQL

### 1. Install PostgreSQL

```bash
./scripts/install_postgresql.sh
```

This runs:

```bash
helm upgrade --install --namespace stock-anomaly-detection openhouse-postgresql bitnami/postgresql -f config/postgresql.yaml
```

**Main configuration in `config/postgresql.yaml`:**

- **Architecture**: `replication` — 1 primary + 1 read replica
- **Credentials**: admin user `postgres` / password `admin`
- **Persistence**: 20Gi on `hostpath` for both primary and read replica
- **Service**: `NodePort` on `30432` (instead of the chart default `ClusterIP`)

> **Databases are created automatically** — `config/postgresql.yaml` bakes a `primary.initdb.scripts` SQL block into the Helm values, which Bitnami's PostgreSQL image runs on its **first** startup against an empty data volume. No manual SQL is needed on a fresh install:

| Database | User | Password | Used by |
|---|---|---|---|
| `keycloak` | `keycloak` | `keycloak` | Keycloak's own auth data |
| `gravitino_db` | `gravitino` | `gravitino` | Gravitino's entity store (metalake/catalog definitions) |
| `iceberg_catalog_db` | `iceberg` | `iceberg` | Iceberg table metadata (schemas/snapshots), used by Gravitino's Iceberg REST service |
| `stock_anomaly` | `stock_user` | `stock_user` | Application OLTP tables — `users`, `user_alert_rules`, `user_alert_events`, `sync_watermarks` |

> ⚠️ `initdb.scripts` only runs once, against a genuinely empty PVC. If PostgreSQL already has data (e.g. after a Helm upgrade, or the PVC survived an uninstall thanks to the `hostpath` Retain policy above), it will **not** re-run and won't create anything new.

### 2. Recreating databases without reinstalling (fallback)

If PostgreSQL already has data and you need `gravitino_db` / `iceberg_catalog_db` (re)created without wiping it:

```bash
./scripts/setup_postgres_databases.sh
```

This connects directly via `kubectl exec` + `psql` and is idempotent (safe to re-run). ⚠️ It only creates **2 of the 4** databases — `gravitino_db` and `iceberg_catalog_db`. It does **not** create `keycloak` or `stock_anomaly`; create those manually if a fresh `initdb` run isn't an option.

#### What's different from the chart's default values?

| Setting | Chart default | This project | Why it was changed |
|---|---|---|---|
| `architecture` | `standalone` | `replication` (1 read replica) | Adds a read replica for the metadata store. |
| `auth.postgresPassword` / `auth.replicationPassword` | `""` (must be supplied) | `"admin"` / `"repl_user"` | Fixed dev credentials. |
| `primary.initdb.scripts` | not set | the 4-database bootstrap SQL described above | Auto-provision every consumer's database/user on first boot. |
| `primary.persistence.storageClass` / `size` | cluster default / (chart default) | `hostpath` / `20Gi` | Pin to this cluster's StorageClass. |
| `primary.service.type` | `ClusterIP` | `NodePort` (`30432`) | Allow connecting to Postgres from outside the cluster for debugging/admin tools. |
| `readReplicas.persistence.storageClass` / `size` | cluster default / (chart default) | `hostpath` / `20Gi` | Same as primary. |

## Keycloak Configuration

Keycloak authenticates two different flows:

- **Gravitino Web UI**: Authorization Code + PKCE (browser-based login)
- **Spark / Trino batch jobs**: Client Credentials (machine-to-machine)

---

**1. Create client `gravitino` for the Gravitino Web UI**

Create a **public** client (Authorization Code + PKCE):

- Client ID: `gravitino`
- Client type: **OpenID Connect**
- Standard flow: **ON**
- Client authentication: **OFF** (public client, no secret)

Tab **Settings**:

- Valid redirect URIs: `https://openhouse.gravitino.test/ui/oauth/callback`
- Web origins: `https://openhouse.gravitino.test`

**2. Create client scope `gravitino` (audience mapper)**

Create a client scope named `gravitino`:

- Protocol: `openid-connect`
- Include in token scope: **ON**

Add an Audience Mapper to this scope:

- Scope `gravitino` → **Mappers** → **Add mapper** → **By configuration** → **Audience**
- Mapper name: `gravitino-audience`
- Included Client Audience: `gravitino`
- Add to access token: **ON**

Assign the scope to client `gravitino`:

- Client `gravitino` → **Client Scopes** → **Add client scope** → select `gravitino`, assigned type **Optional**

**3. Create client `spark` for Spark/Trino jobs**

Create a confidential client:

- Client ID: `spark`
- Client authentication: **ON**

After creation, get the **client secret** from the Credentials tab.

**4. Create client scope `sign` for `spark`**

Create a client scope named `sign` (include in token scope: **ON**), then add it to client `spark` as **Default**.

**5. Assign the `gravitino` scope to the `spark` client as Default (required for Iceberg REST)**

This is required for Gravitino's Iceberg REST service (port `9001`) to authenticate with the Gravitino API (port `8090`) via dynamic catalog config. The `spark` client obtains a token with `scope=gravitino`; that token must contain `aud: gravitino` (added by the Audience Mapper on the `gravitino` scope) or the Gravitino API returns `401` → `ServiceFailureException`.

> **Why Default, not Optional?** Client Credentials flow (machine-to-machine) has no user to request scopes at login time — only **Default** scopes are included automatically. **Optional** scopes only get included when explicitly requested in an Authorization Code flow.

- Client `spark` → **Client Scopes** → **Add client scope** → select `gravitino`, assigned type **Default**

**6. Remove the `roles` scope from client `spark` (fix the JWT `aud` claim)**

Gravitino's `JwksTokenValidator` requires `aud` to be exactly `"gravitino"` (a single string). Keycloak's built-in `roles` scope automatically adds `account` to `aud`, producing `["gravitino", "account"]`, which causes:

```
JWKS JWT validation error: JWT aud claim has value [gravitino, account], must be [gravitino]
```

Fix: Client `spark` → **Client Scopes** → find `roles` under **Assigned Default Client Scopes** → **Remove**.

After this, tokens issued for `spark` will have `"aud": "gravitino"` as a single value, and Gravitino will accept them.

> **Verify**: request a token with `grant_type=client_credentials&client_id=spark&scope=gravitino`, decode the JWT payload, and confirm `aud` is a plain string, not an array.

**7. Create a user**

Users → Create new user: username `admin`, password `admin`, Temporary: **OFF**.

## Installing Gravitino

Gravitino is the Iceberg REST catalog — both Spark (batch writes) and Trino (queries) talk to it to resolve table metadata.

### Before you start

- **Keycloak** must already be running and configured (previous section) — Gravitino authenticates every request through it.
- **PostgreSQL** must already have `gravitino_db` and `iceberg_catalog_db` (created automatically by `install_postgresql.sh`, see above).
- **MinIO** must already be running — this is where the actual Iceberg data files live.

### 1. Create the TLS secrets

```bash
./scripts/create_secret_gravitino_tls_ingress.sh
```

Generates a self-signed cert for `openhouse.gravitino.test` and creates the `gravitino-tls` secret for Gravitino's own ingress.

```bash
./scripts/create_secret_gravitino_tls.sh
```

Imports Keycloak's CA certificate (`tls/keycloak_tls.cert`) into the `keycloak-ca-cert` secret, so Gravitino's JVM trusts Keycloak's self-signed HTTPS certificate when validating tokens. Run `create_secret_keycloak_tls.sh` (Keycloak section above) first, or this will fail with a missing-file error.

### 2. Install Gravitino

```bash
./scripts/install_gravitino.sh
```

This runs:

```bash
helm upgrade --install --namespace stock-anomaly-detection openhouse-gravitino helm/gravitino -f config/gravitino.yaml
```

**Main configuration in `config/gravitino.yaml`:**

- **Entity store**: PostgreSQL, `gravitino_db`
- **Auth**: OAuth2/OIDC against Keycloak realm `iceberg` (issuer `https://openhouse.keycloak.test/realms/iceberg`)
- **Catalog mode**: `dynamic-config-provider`, metalake `stock_metalake` — catalogs (warehouses) are created at runtime via the Gravitino API, not baked into Helm values
- **Iceberg REST OAuth2**: `additionalConfigItems` configure the `spark` client credential so the Iceberg REST service (port `9001`) can itself authenticate as an OAuth2 client against Keycloak
- **JVM truststore**: mounted from the `keycloak-ca-cert` secret, so the JVM trusts Keycloak's HTTPS cert
- **Ingress**: HTTPS at `openhouse.gravitino.test`, TLS via `gravitino-tls`

> ⚠️ `scripts/template_gravitino.sh` renders from the **public** OCI chart `oci://registry-1.docker.io/bitnamicharts/gravitino`, while `scripts/install_gravitino.sh` deploys the **local vendored copy** in `helm/gravitino/`. If that local copy has been modified, the rendered preview won't match what's actually installed.

#### What's different from the chart's default values?

`config/gravitino.yaml` overrides the vendored chart's own default `helm/gravitino/values.yaml`. Unlike MinIO/PostgreSQL/Keycloak, this isn't a full copy-with-overrides — it's a smaller, targeted set of changes on top of the chart defaults:

| Setting | Chart default | This project | Why it was changed |
|---|---|---|---|
| `entity.jdbcUrl` / `jdbcDriver` | `jdbc:h2` / `org.h2.Driver` (embedded, ephemeral) | `jdbc:postgresql://openhouse-postgresql-primary:5432/gravitino_db` / `org.postgresql.Driver` | Persist Gravitino's own metadata (metalakes, catalog definitions) in the shared PostgreSQL instead of an embedded H2 DB that's lost on pod restart. |
| `icebergRest.catalogConfigProvider` / `dynamicConfigProvider` | unset (static `catalogBackend: memory`, `warehouse: /tmp/`) | `dynamic-config-provider`, pointing at `http://localhost:8090` (self), metalake `stock_metalake` | Let the Iceberg REST service (port `9001`) read catalog config live from the Gravitino API, so warehouses can be added/removed without a Helm upgrade — see "Setting up the Metalake and Warehouses" below. |
| `authenticators` | `simple` (no auth) | `oauth` | Require OAuth2 authentication on every request. |
| `authenticator.oauth.*` | `serviceAudience: test`, empty `serverUri`, placeholder `tokenPath` | Full Keycloak JWKS config — `provider: oidc`, `clientId: gravitino`, `authority: https://openhouse.keycloak.test/realms/iceberg`, `jwksUri` pointing at the in-cluster Keycloak service, `tokenValidatorClass: JwksTokenValidator` | Validate JWTs against Keycloak's live JWKS endpoint (auto-picks up key rotation) instead of a static signing key. |
| `additionalConfigItems` | `{}` | `gravitino.iceberg-rest.gravitino-auth-type=oauth2` + `gravitino-oauth2.credential=spark:<secret>` + trusted-issuer `authorities` list | Required so the Iceberg REST service can itself authenticate as an OAuth2 client when it calls the Gravitino API internally — without this, the API returns `401` and Iceberg REST fails with `ServiceFailureException`. |
| `extraVolumes` / `extraVolumeMounts` | only `gravitino-log` | adds `aws-libs` (AWS SDK/S3FileIO JAR, populated by an init container), `truststore` (custom JVM truststore), `keycloak-ca-cert` (mounted from the secret created by `create_secret_gravitino_tls.sh`) | Support S3/MinIO access via `S3FileIO` and trusting Keycloak's self-signed HTTPS cert. |
| `extraInitContainers` | none | 3 added: schema init against the external PostgreSQL, download the `gravitino-iceberg-aws-bundle` JAR from Maven Central, import the Keycloak CA cert into a JVM truststore | Bitnami's chart only auto-runs schema init when using its *bundled* PostgreSQL subchart (`postgresql.enabled: true`) — since this project uses an external Postgres, that step has to be replicated manually. The other two init containers support the S3FileIO JAR and HTTPS trust, respectively. |
| `env.GRAVITINO_MEM` | `-Xms1024m -Xmx1024m -XX:MaxMetaspaceSize=512m` | same, plus `-Djavax.net.ssl.trustStore=/opt/gravitino-truststore/gravitino.jks` (+ password/type) | Point the JVM at the custom truststore above so it trusts Keycloak's cert when fetching OIDC discovery/JWKS over HTTPS. |
| `env.GRAVITINO_CLASSPATH` | not set | `/aws-ext-libs/gravitino-iceberg-aws-bundle-1.1.0.jar` | Add the downloaded AWS bundle JAR to the classpath — fixes `NoClassDefFoundError: software/amazon/awssdk/...`. |
| `ingress.enabled` / `hosts` / `tls` | `false` / `chart-gravitino.local` / none | `true` / `openhouse.gravitino.test` / `gravitino-tls`, plus `nginx` proxy annotations with a 600s timeout | Expose Gravitino outside the cluster over HTTPS; the long timeout accommodates slower metadata operations. |
| `resources` | not set | requests `200m`/`512Mi`, limits `1000m`/`2Gi` | Bound resource usage instead of unlimited. |

> The chart default also leaves `persistence.storageClassName` unset; the project sets it to `standard-rwo` (a GKE-style class name that doesn't exist in this cluster). This has **no effect** because `persistence.enabled` is `false` for both — Gravitino itself is stateless, all real state lives in PostgreSQL and MinIO. Harmless leftover, not worth fixing unless persistence is ever enabled.

### 3. Preview / Uninstall

```bash
./scripts/template_gravitino.sh
./scripts/uninstall_gravitino.sh
```

`uninstall_gravitino.sh` also deletes Gravitino's PVCs by label — a plain `helm uninstall` alone would leave them behind.

### 4. Setting up the Metalake and Warehouses (Iceberg catalogs)

Before any Spark or Trino job can write/read Iceberg tables, the metalake `stock_metalake` and its catalogs must exist in Gravitino.

**Option A — manually via the UI:**

1. Go to `https://openhouse.gravitino.test/ui`, click **Create Metalake**, name it `stock_metalake` (must match `dynamicConfigProvider.metalake` in `config/gravitino.yaml` exactly, or every Spark/Trino request to the Iceberg REST endpoint fails with `metalake not found`).
2. Inside the metalake, go to **Catalogs** → **Create Catalog**, type `Relational`, provider `Apache Iceberg`, and set these properties:

```text
catalog-backend      = jdbc
uri                  = jdbc:postgresql://openhouse-postgresql-primary:5432/iceberg_catalog_db
warehouse            = s3://bronze/warehouse

jdbc-driver          = org.postgresql.Driver
jdbc-user            = iceberg
jdbc-password        = iceberg
authentication.type  = simple

io-impl              = org.apache.iceberg.aws.s3.S3FileIO
s3-endpoint          = http://openhouse-minio:9000
s3-region            = us-east-1
s3-path-style-access = true
s3-access-key-id     = admin
s3-secret-access-key = admin123
```

Repeat for `silver` and `gold`, changing `warehouse` to `s3://silver/warehouse` / `s3://gold/warehouse`. Make sure the `bronze`/`silver`/`gold` buckets already exist in MinIO.

**Option B — via script:**

```bash
./scripts/setup_gravitino_warehouses.sh
```

Calls the Gravitino REST API directly to create the metalake and catalogs. Its `METALAKE_NAME` default is `stock_metalake`, matching `config/gravitino.yaml`. ⚠️ It only creates the `bronze` and `silver` warehouses — not `gold`, even though Trino's configuration (`compute/config/trino.yaml`) expects all three catalogs to exist. Create the `gold` catalog manually (Option A) if using this script.
